## 1. 소프트웨어 개발 프로세스의 이해

### 프로세스란?

- **정의**: 일을 처리하는 과정 또는 순서
  - 예) 공장에서 자동차가 조립되어 완제품이 되는 과정
  - 예) TV 요리 프로에서 요리사가 요리를 만드는 과정

### 프로세스의 목적

- 이전에 얻은 **노하우 전달** → 시행착오 감소 → 빠르게 적응
- **가이드** 역할

### 프로세스의 3요소

```
능력/교육/동기가 부여된 인원 + 프로세스 + 도구와 장비
```

## 2. 소프트웨어 프로세스 모델의 이해

### SDLC (Software Development Life Cycle)

- SW를 **어떻게 개발할 것인가**에 대한 전체적인 흐름을 체계화한 개념
- 개발 계획 수립부터 최종 폐기까지의 **전 과정**을 다룸
- 순차적인 단계로 이루어짐

### 소프트웨어 프로세스 모델의 목적

- 소프트웨어 개발의 전 과정을 하나의 프로세스로 정의
- 주어진 예산과 자원으로 개발하고 관리하는 방법을 구체적으로 정의
- **고품질**의 소프트웨어 제품 생산

## 3. 주먹구구식 모델

> **Build and Fix 모델** / Code and Fix 모델 / 즉흥적 소프트웨어 개발 모델

### 개념

- 공식적인 가이드라인이나 프로세스가 **없는** 개발 방식
- 요구 분석 명세서나 설계 단계 없이 간단한 기능만 정리하여 개발
- 코드 먼저 작성 → 문제 수정 반복

### 개발 단계

```
코딩(구현) → 제품 → 문제? --[아니오]--> 사용
                  ↑         |
                  └--[예]--수정
```

### 사용 적합 상황

- 개발자 **한 명**이 단시간에 마칠 수 있는 경우
- 대학 수업의 한 학기용 프로젝트 정도

### 단점

| 문제 | 설명 |
|------|------|
| 관리 어려움 | 문서화된 산출물이 없어 유지보수 곤란 |
| 아키텍처 부재 | 좋은 구조를 만들 수 없음 |
| 협업 불가 | 일을 나눠 개발하거나 진척 파악 불가 |
| 구조 악화 | 계속적 수정으로 프로그램 구조가 점점 나빠짐 |

## 4. 선형 순차적 모델 (폭포수 모델)

> **Linear Sequential 모델** / Waterfall 모델 / Classic Life Cycle

### 개발 단계

```
계획 → 요구 분석 → 설계 → 구현 → 테스트 → 유지보수
 ↕         ↕        ↕      ↕        ↕
검증       검증     검증   검증     검증
```

각 단계 산출물:
- 계획 → `개발 계획서`
- 요구 분석 → `요구 사항 정의서`, `요구 분석 명세서`
- 설계 → `설계 사양서`
- 구현 → `모듈별 코드`

### 장점

- 관리의 **용이성**
- **체계적인 문서화**
- 요구사항의 변화가 적은 프로젝트에 적합

### 단점

- 각 단계는 앞 단계가 **완료되어야** 수행 가능
- 각 단계 결과물이 완벽해야 다음 단계에 오류를 넘기지 않음
- 사용자가 중간에 **가시적인 결과를 볼 수 없음**

## 5. V 모델

> 폭포수 모델 + **테스트 단계 추가 확장**

### 폭포수 모델 vs V 모델

| 구분 | 폭포수 모델 | V 모델 |
|------|------------|--------|
| 초점 | 산출물 중심 | 각 개발 단계 **검증** 중심 |

### V 모델 구조

```
요구 분석 ←─────── 요구 분석 검증 ──────→ 시스템 테스트 → 인수 테스트
    ↓                                           ↑
아키텍처 설계 ←── 인터페이스 검증 ──→ 통합 테스트
    ↓                                    ↑
  모듈 설계 ←───── 모듈 검증 ──────→ 단위 테스트
    ↓                                    ↑
              구현 ─────────────────────┘
```

## 6. 진화적 프로세스 모델

> 선형순차적 모델(폭포수)의 한계를 극복하기 위해 등장

### 프로토타입 모델

**프로토타입이란?**
- 대량 생산에 앞서 미리 제작해보는 **원형/시제품**
- 사용자의 요구를 받아 일단 모형을 만들고, 사용자와 **의사소통하는 도구**로 활용

**개발 생명주기 (프로토타입 모델)**

```
요구 수정 → 빠른 설계 → 프로토타입 개발 → 고객 평가 → 프로토타입 조정 → 구현
```

### 실험적 프로토타입 모델

```
요구 분석 → 빠른 설계 → 프로토타입 개발 → 고객 평가
                ↑                              |
                └──── 수정 요구 사항 ←──[아니오]┘
                                               |[예]
                                    최종 사용자 요구 → 분석 → 설계 → 구현
```

### 진화적 프로토타입 모델

```
요구 분석 → 빠른 설계 → 프로토타입 개발 → 평가?
                ↑                            |[아니오]
                └────── 수정 요구 사항 ←────┘
                                             |[예]
                                          최종 제품
```

### 프로토타입 개발 단계

1. **요구사항 정의 및 분석**: 1차 개략적 요구 정의 후 n차 반복
2. **프로토타입 설계**: 사용자와 대화 가능한 수준의 설계, UI 중심
3. **프로토타입 개발**: 완제품이 아닌 요구 항목 확인용 모형

## 7. 통합 프로세스 모델 (UP)

> 폭포수 모델의 문제점 → **반복적 개발 방법**으로 해결

### 반복적 개발 방법

```
R₁→A₁→D₁→C₁→T₁→M₁ ┐
                      └→ R₂→A₂→D₂→C₂→T₂→M₂ ┐
                                               └→ R₃→A₃→...

R: 요구사항 정의 | A: 분석 | D: 설계 | C: 구현 | T: 테스트 | M: 유지보수
```

### UP 모델의 4단계

| 단계 | 영문 | 설명 |
|------|------|------|
| 도입 | Inception | 프로젝트 초기, 비즈니스 모델링/요구사항 집중 |
| 구체화 | Elaboration | 분석·설계 집중, 구체화 1~2단계 반복 |
| 구축 | Construction | 구현·테스트 집중, 구축 1~n단계 반복 |
| 전이 | Transition | 배포·전이, 전이 1~2단계 반복 |

### 공통 작업 (전 단계에 걸쳐 수행)

- 형상(변화) 관리
- 프로젝트 관리
- 환경 점검

### 비즈니스 모델링

- 소프트웨어 개발 이전에 **비즈니스 환경을 이해/분석**하는 핵심 작업
- 프로젝트 초기(도입/구체화 단계)에 집중
- 주요 역할:
  - 시스템 목적 및 범위 정의
  - 현실 세계의 개념화 (도메인 모델 구축)
  - 업무 프로세스 분석 (BPMN 활용)
  - 이후 요구사항 정의 및 설계 단계의 **기반 제공**

### BPMN (Business Process Model and Notation)

- 조직 내의 **업무 흐름(비즈니스 프로세스)을 시각화**하고 표준화하기 위한 글로벌 그래픽 표기법

## 8. 애자일 프로세스 모델

### 애자일이란?

- 의미: **"날렵한", "민첩한"**
- 고객의 요구에 민첩하게 대응하고 그때그때 주어지는 문제를 풀어나가는 방법론

### 애자일 선언문 (4가지 핵심 가치)

| ❌ 기존 방식 | ✅ 애자일 방식 |
|------------|-------------|
| 프로세스와 도구 중심 | 개개인과의 상호 소통 중시 |
| 문서 중심 | 실행 가능한 소프트웨어 중시 |
| 계약과 협상 중심 | 고객과의 협력 중시 |
| 계획 중심 | 변화에 대한 민첩한 대응 중시 |

### 애자일 개발 방법

- **반복적인 개발**을 통한 잦은 출시를 목표

```
프로토타입 개발 → 사용자 확인 → 일부 기능 사용
```

### 스크럼 (Scrum)

> "팀원들이 럭비의 스크럼처럼 똘똘 뭉쳐서(Scrum), 짧은 구간을 전력 질주(Sprint)하며 목표를 달성한다"

**스크럼의 특징**

- 소프트웨어 개발보다는 **팀의 개선과 프로젝트 관리**를 위한 애자일 방법론
- 경험적 관리 기법 중 하나
- 구체적인 프로세스를 명확하게 제시하지 않음
- 개발 팀을 운영하는 **효율적인 운영 방식(지침)**

**스크럼 진행 과정**

```
① 제품 기능 목록 작성 (Product Backlog)
        ↓
② 스프린트 계획 회의 → 스프린트 구현 목록
        ↓
③ 스프린트 수행 (1~4주)
   - 매일 일일 스크럼 회의
   - 스크럼 마스터 지원
   - 소멸 차트로 진척 관리
        ↓
④ 스프린트 개발 완료 → 최종 제품
        ↓
⑤ 스프린트 검토 회의 + 스프린트 회고
```

## 📋 모델 비교 요약

| 모델 | 특징 | 적합한 상황 |
|------|------|-----------|
| 주먹구구식 | 계획 없이 코딩 → 수정 반복 | 1인 소규모, 단기 프로젝트 |
| 폭포수 | 순차적, 단계별 완료 후 진행 | 요구사항 변화가 적은 프로젝트 |
| V 모델 | 폭포수 + 각 단계 검증 강화 | 품질 검증이 중요한 프로젝트 |
| 프로토타입 | 시제품 반복 제작 → 요구 확정 | 요구사항이 불명확한 경우 |
| 통합(UP) | 반복적 + 단계적 개발 | 대규모, 객체지향 프로젝트 |
| 애자일(스크럼) | 짧은 스프린트 반복, 고객 소통 | 변화가 잦고 빠른 출시 필요 |

---

> 📝 **요즘 소프트웨어 개발에 사용하는 방법론?**
> 현업에서는 **애자일(특히 스크럼)** 이 가장 많이 사용됨.
> 대규모 엔터프라이즈 프로젝트에서는 UP 또는 폭포수와 애자일을 혼합한 방식도 활용.
